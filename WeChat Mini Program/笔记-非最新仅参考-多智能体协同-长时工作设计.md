# 笔记 - 微信小程序多智能体协同 - 长时工作设计

> 仅供参考的设计思考笔记，记录微信小程序场景下多智能体协同的核心决策与权衡。最新可执行规范以《主智能体提示词.md》和 `agents/` 下子智能体定义为准。

---

## 一、核心原则

### 1.1 Harness Engineering 的本质

不是让 AI 一次做完整套小程序，而是设计一套"套件(harness)"——决定存储什么、检索什么、展示给模型什么。主智能体就是这个套件的编排者。

**核心矛盾**：上下文窗口有限 vs 项目复杂度无限（一个完整的小程序可能有 20+ 页面、30+ 组件、API 封装、状态管理）。解法不是更大的窗口，而是更聪明的上下文管理。

### 1.2 三条铁律

| 铁律 | 含义 | 违反后果 |
|------|------|----------|
| **文件即记忆** | 子智能体的所有产出必须持久化到 `.wxml/.wxss/.js/.json` 文件，不依赖内存 | 上下文丢失后无法恢复 |
| **隔离即常态** | 每个子智能体只看到主智能体给它的信息（如目标文件路径、规范、需求段落） | 隐式依赖导致幻觉，比如错误使用未注册的 component |
| **记录即保险** | 主智能体和子智能体都写日志，包括 `app.json` 的修改痕迹 | 出问题后无法复盘哪个 Agent 引入了配置漂移 |

## 二、角色体系设计

### 2.1 三角色模型 + 三测试维度

```
┌─────────────┐
│   主智能体    │  编排者：拆任务、管状态、中转信息
│  (Orchestrator) │
└──────┬──────┘
       │
   ┌───┴────────────────────┐
   │           │            │
   ▼           ▼            ▼
┌──────────┐  ┌──────────┐  ┌──────────────────────┐
│  规划智能体 │  │ 开发智能体 │  │   三个并行测试智能体    │
│ (Planner)  │  │  (Dev)   │  │  ─ 结构规范 (Structure) │
└──────────┘  └──────────┘  │  ─ 样式交互 (Style)     │
                              │  ─ 性能安全 (Performance)│
                              └──────────────────────┘
```

**各角色核心职责**：

| 角色 | 职责 | 工具权限 | 上下文管理 |
|------|------|----------|------------|
| **主智能体** | 拆任务、管计划、中转信息、写日志、不写代码 | 全部 | 保持全程上下文，但严格不读子Agent产物 |
| **规划智能体 wmp-planner** | 读需求 → 拆页面/组件 → 搭骨架（app.json/app.js/app.wxss/utils）| Read + Write + Bash | 一次性运行，结束即释放 |
| **开发智能体 wmp-dev** | 读需求 → 写 4 件套 → 自测 → 修正 | Read + Write + Edit + Bash + Glob + Grep | 每批次新创建或 resume |
| **结构测试 wmp-tester-structure** | 审查 WXML 节点、目录结构、配置文件 | Read + Glob + Grep + Write（只写报告） | 每批次新创建或 resume |
| **样式测试 wmp-tester-style** | 审查 WXSS 规范、rpx、组件使用、可访问性 | 同上 | 同上 |
| **性能测试 wmp-tester-performance** | 审查 setData、网络请求、存储、安全 | 同上 | 同上 |

### 2.2 为什么是三个测试维度而不是一个

小程序的"质量"由不同侧面构成，且不同侧面的检查标准差异很大：

- **结构规范**：是否符合微信小程序的项目结构与文件约定（pages 数组、usingComponents、目录命名、WXML 节点上限）
- **样式与交互**：WXSS 是否使用 rpx、组件是否使用微信内置组件而非 HTML 标签、bindtap 是否合理、可访问性属性
- **性能与安全**：setData 调用频率、单次 setData 数据量、wx.request 是否 HTTPS、本地存储是否泄漏敏感信息

把这三个维度分开，每个测试 Agent 拥有自己的检查清单，**互不干扰，且可以并行运行**（主Agent 同时启动 3 个，等三个都回报）。

### 2.3 为什么把 page 和 component 区分批次

虽然 page 和 component 的文件组成相同（都是 4 件套），但开发上下文差异显著：

- **page** 关注路由、生命周期（onLoad/onShow/onReady/onHide/onUnload）、与 `app.json` 联动
- **component** 关注 properties/data/observers/behaviors、生命周期（created/attached/detached）、`triggerEvent` 与父子通信

混合开发会导致 wmp-dev 频繁切换上下文，且修改 `app.json` 与 `usingComponents` 的逻辑混在一起容易出错。

## 三、工作流程

### 3.1 整体流程图

```
开始
  │
  ▼
[主智能体] 收集需求文档路径、输出目录、批量大小
  │
  ▼
[wmp-planner] 阅读需求 → 产出 dev-plan.md + page-spec.md + 搭建骨架
  │
  ▼
═══════════ 批量循环（按 BATCH_SIZE 分组）═══════════
  │
  ├─► [wmp-dev] 新建，开发本批所有 page/component
  │     ├─ 读取：需求段落、page-spec、lessons-learned、已有代码
  │     ├─ 写 4 件套文件
  │     ├─ page 时同步追加到 app.json/pages
  │     └─ 输出：完成确认
  │
  ├─► [三个测试Agent并行] 新建，分别审查本批所有任务
  │     ├─ wmp-tester-structure：节点数、目录结构、配置正确性
  │     ├─ wmp-tester-style：WXSS rpx、组件使用、可访问性
  │     └─ wmp-tester-performance：setData、网络请求、存储、安全
  │
  ├─► 三维结果汇总？
  │     ├─ 全PASS → 进入下一批
  │     └─ 任一FAIL → 修正循环 ↓
  │
  ├─► [wmp-dev] resume，一次性修正所有 FAIL 维度
  │
  ├─► [对应测试Agent] resume，重测本批所有任务
  │
  └─► 回到"三维结果汇总"判断（最多 3 轮）
═══════════════════════════════════
  │
  ▼
全部批次完成 → 整体配置校验 → 报告完成
```

### 3.2 修正循环

```
修正循环第 K 轮（K ≤ 3）：

1. resume 开发智能体：
   Agent(
     resume: "{DEV_ID}",
     subagent_type: "wmp-dev",
     prompt: "
       请读取以下测试报告并修正所有问题：
       {所有 FAIL 维度的报告路径}

       目标文件：{FAIL 任务对应的 wxml/wxss/js/json 路径}
       简短确认即可。
     "
   )

2. resume 仍 FAIL 维度的测试智能体（不需要 PASS 维度重测）

3. 判定：
   - 全 PASS → 退出
   - 仍有 FAIL 且 K < 3 → 继续
   - K = 3 仍 FAIL → 强制通过，标记"⚠️ 低质量通过"
```

## 四、关键操作细节

### 4.1 子智能体 ID 获取

每次创建子智能体后，立即获取其真实 ID：

```bash
find ~/.claude/projects/ -name "agent-*.meta.json" -type f -printf '%T@ %p\n' 2>/dev/null | sort -rn | head -1 | cut -d' ' -f2-
```

从文件名提取：`agent-a95e84cd0b54c85ad.meta.json` → 裸 ID = `a95e84cd0b54c85ad`

### 4.2 Resume 关键要点

- `resume` 参数填**裸 ID**，不是完整文件名
- `subagent_type` 必须与创建时一致
- resume 后子智能体能看到完整的历史对话（包括上次的产出与测试结果）

### 4.3 后台 Agent 与超时

测试 Agent 都用 `run_in_background: true` 启动，原因是：

- 三个测试维度并行，无需等单个完成
- 完成时系统自动通知主Agent
- 避免 TaskOutput 阻塞主Agent上下文

如果 TaskOutput 超时（300s），改用 Grep 提取第一行判定：

```bash
Grep(pattern="^### 判定", path="{OUTPUT_DIR}/test-reports/{NAME}-structure.md", head_limit=1)
```

## 五、上下文管理策略

### 5.1 每个子智能体必须读取的上下文

**wmp-dev 的必读清单**（开发模式）：

```
1. 任务描述（主智能体传入的 prompt）         → 知道做什么 page/component
2. page-spec.md 中对应的段落                  → 知道做成什么样
3. lessons-learned.md                          → 知道别人踩过什么坑
4. app.json                                    → 知道全局配置（如 tabBar、permission）
5. 同类型已完成的 page/component（1-2 个）    → 保持风格一致
6. 该 page/component 引用的组件（如有）        → 知道 usingComponents 关系
```

**wmp-tester-* 的必读清单**：

```
1. 任务描述（主智能体传入的 prompt）         → 知道测什么
2. page-spec.md 中对应的段落                  → 知道验收标准
3. 待测的 4 件套文件                          → 审查对象
4. 对应的检查清单（在 agent 定义中）          → 知道用什么标准
```

### 5.2 经验库设计

经验库 `lessons-learned.md` 是跨批次传递知识的唯一可靠通道。

**良好经验示例**：

- ✅ "wxss 中不应使用 ID 选择器、属性选择器和标签名选择器（因为组件样式隔离规则限制）"
- ✅ "wx:for 列表项必须指定 wx:key，避免数据更新时整列表重渲染"
- ✅ "wx.request 必须封装统一的 token 拦截器，避免每个页面重复处理 401"

**反例**：

- ❌ "pages/profile/profile.js 第 42 行的 setData 调用要合并" → 太具体，只适用一个文件
- ❌ "注意性能问题" → 没有可操作性

判断标准：如果一条经验能帮到下一个子智能体避免类似错误，就是好的粒度。

### 5.3 开发计划文件

**文件路径**：`{OUTPUT_DIR}/dev-plan.md`

**示例格式**：

```markdown
# 开发计划

## 项目信息
- 需求文档：{SPEC_FILE}
- 总任务数：{N}（页面 {P} 个 / 组件 {C} 个）
- 创建时间：{时间}

## 任务清单

| # | 类型     | 标识        | 标题            | 状态 | 备注 |
|---|----------|------------|----------------|------|------|
| 0 | -        | -          | 公共基础        | ✅   | 计划Agent直接完成 |
| 1 | page     | index      | 首页            | ⏳   | tabBar 第一项 |
| 2 | page     | profile    | 个人中心        | ⏳   | tabBar 第二项 |
| 3 | page     | detail     | 商品详情        | ⏳   | 从首页跳入 |
| 4 | component| product-card | 商品卡片组件  | ⏳   | 首页和详情都用 |
| 5 | component| price-tag  | 价格标签组件    | ⏳   | 嵌在 product-card 内 |

状态： ⏳ 待办 | 🔄 进行中 | ✅ 完成 | ⚠️ 低质量通过
```

## 六、子智能体模板设计

### 6.1 模型选择原则

| 场景 | 推荐模型 | 理由 |
|------|----------|------|
| wmp-planner | inherit | 需要全局架构能力 |
| wmp-dev | inherit | 开发需要最强推理 |
| wmp-tester-structure | haiku | 检查清单式工作，快速便宜 |
| wmp-tester-style | haiku | 同上 |
| wmp-tester-performance | haiku | 同上 |

### 6.2 权限设计

| 角色 | 工具权限 | 理由 |
|------|----------|------|
| **wmp-planner** | Read + Write + Bash + Glob + Grep | 需要搭骨架、创建目录 |
| **wmp-dev** | Read + Edit + Write + Bash + Glob + Grep | 需要读写代码、运行命令 |
| **wmp-tester-*** | Read + Write + Glob + Grep | 只读代码，Write 仅用于写测试报告 |
| **wmp-tester-* 禁止** | Edit | 防止测试智能体直接修改代码（Write 受路径约束在 test-reports/ 下） |

## 七、开发计划拆分策略

### 7.1 拆分粒度

| 太粗 | 恰好 | 太细 |
|------|------|------|
| "做完整个小程序" | "开发 pages/profile 个人中心页" | "写 profile.js 第 12 行的 setData" |
| 一个任务 > 5 个文件 | 一个 page = 4 件套 = 一个任务 | 一个任务 < 1 个完整文件 |
| 难以测试 | 可独立验证 | 上下文切换成本太高 |

**黄金法则**：一个子任务 = 一个 page 或一个 component（即一组 4 件套）。

### 7.2 拆分顺序

```
Phase 0: wmp-planner 搭骨架
  → app.json / app.js / app.wxss / project.config.json / sitemap.json
  → utils/ 工具目录（如 request.js / storage.js）
  → 空的 pages/ 和 components/ 目录

Phase 1: 优先开发被依赖的 component
  → 例如 product-card 组件被多个 page 用到，必须先于这些 page 开发

Phase 2: 开发 page，按用户流走（启动页 → tabBar 页 → 二级页 → 弹窗页）

Phase 3: 整体配置校验
  → app.json/pages 列表完整
  → tabBar 路径都存在
  → sitemap.json 配置合理
```

### 7.3 依赖关系处理

- 如果 page A 依赖 component B，B 必须先完成并通过测试
- utils 类（request、storage、format）建议在 Phase 0 由 wmp-planner 直接产出
- 如果发现新依赖（如 page 开发到一半需要新组件），暂停当前批，先完成依赖项

## 八、常见问题与踩坑指南

### 8.1 微信小程序特有的坑

| 问题 | 原因 | 解法 |
|------|------|------|
| 真机 selectComponent 返回 null | 在 onLoad 中调用，DOM 未渲染 | 改在 onReady 或 attached 中调用 |
| WXSS 文件名/标签选择器不生效 | 组件样式隔离限制 | 改用 class 选择器 |
| wx.setData 异步存值后立即取出为空 | 同步取异步存的数据 | 用 setStorageSync 或在 success 回调中取 |
| 真机 wx.request 失败 | 未在小程序后台配置合法域名 | 测试时勾选"不校验合法域名"，生产前配置 |
| this.data 修改后视图不更新 | 直接赋值未触发响应式 | 必须用 this.setData() |
| tabBar 配置后页面不显示 | tabBar 页面必须在 app.json/pages 中 | 同步更新 pages 数组 |
| 组件无法接收父组件传入的对象更新 | 引用没变 | properties 中用浅拷贝或新引用触发 observer |

### 8.2 流程类问题

| 问题 | 原因 | 解法 |
|------|------|------|
| Agent(resume) 找不到 ID | 路径不对，有双层子目录 | 用 find 命令的标准方案 |
| Resume 报 "Agent type not found" | 未指定 subagent_type | 必须指定与创建时一致的 subagent_type |
| 测试 Agent 误判 PASS | 检查标准不够明确 | 在 Agent 定义中给出阈值和反例 |
| 经验库太长，子智能体读不完 | 积累了太多条目 | 定期精简，只保留通用的、高价值的 |
| 开发智能体重复踩坑 | 没读经验库 | 在 prompt 中强制要求先读 lessons-learned.md |

## 九、适用场景与不适用场景

### 适用场景

- 中大型小程序（10+ 页面、10+ 组件）
- 需要严格代码规范的商业项目
- 需要反复迭代验证的复杂业务
- 团队协作模拟（开发 + 审查分工）

### 不适用场景

- 1-3 页面的 demo 小程序（一个智能体就够）
- 完全偏后端的 cloud function 项目
- 不需要质量验证的原型

## 十、与其他方法的对比

| 维度 | 单智能体长对话 | 双窗口手动切换 | 本方法论（主智能体编排） |
|------|---------------|---------------|----------------------|
| 上下文管理 | 差（窗口爆满） | 中（手动管理） | 好（自动隔离+传递） |
| 迭代效率 | 低（全在对话中） | 中（手动传递） | 高（自动循环 + 并行测试） |
| 可追溯性 | 差（对话太长） | 中（两份记录） | 好（main-log + 三份测试报告） |
| 经验积累 | 差（对话结束即丢失） | 差（依赖人工总结） | 好（lessons-learned 自动累积） |
| 实施复杂度 | 低 | 中 | 高（需要精心设计模板） |

## 十一、快速启动清单

开始一个新小程序项目时，按以下清单准备：

```
□ 1. 准备需求文档（产品需求.md / 接口文档.md / UI 原型说明.md）
□ 2. 选定输出目录（空目录或新建目录）
□ 3. 拷贝本 skill 到 .claude/skills/wechat-miniprogram/
□ 4. 拷贝 agents/*.md 到 .claude/agents/
□ 5. 向主智能体说明：
     - 需求文档路径
     - 输出目录
     - 批量大小（默认 1，可指定 2-3）
□ 6. 主智能体自动启动 wmp-planner
□ 7. 逐批执行开发-测试循环
□ 8. 最后整体配置校验
□ 9. 用微信开发者工具打开项目验证
□ 10. 填入真实 appid，配置 request 合法域名
```
